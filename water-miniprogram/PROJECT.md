## 七、业务流程

### 7.1 购物流程
```
首页/详情页 → 加入购物车 → 购物车 → 确认订单 → 支付 → 待配送 → 配送中 → 完成
```

### 7.2 订单状态流转
```
待支付 (pending)
    ↓ 支付
待配送 (paid)
    ↓ 配送
配送中 (delivering)
    ↓ 确认收货
已完成 (completed)

待支付 → 取消 → 已取消 (cancelled)
```

## 十、首页 UI 优化记录

> 2026-03-19 完成

### 10.1 优化内容

#### 布局重构
- **海报区**：从全屏改为 50vh（最小 400rpx，最大 600rpx）
- **商品区**：独立白色背景区域，标题「在销商品」
- **图片显示**：使用 `mode="widthFix"` 显示图片上半部分

#### 过渡效果（Glassmorphism 毛玻璃）
采用三层叠加结构实现自然过渡：

```
海报层 (z-index: 1)
    ↓
毛玻璃过渡层 (z-index: 3)  ← backdrop-filter: blur(30rpx)
    ↓
商品区 (z-index: 4)  ← 白色卡片叠加效果
```

#### 关键样式
```css
.hero-transition {
  background: rgba(255, 255, 255, 0.85);
  backdrop-filter: blur(30rpx);
  -webkit-backdrop-filter: blur(30rpx);
  border-top-left-radius: 48rpx;
  border-top-right-radius: 48rpx;
  box-shadow: 0 -8rpx 32rpx rgba(0, 0, 0, 0.08);
}

.product-section {
  margin-top: -60rpx;
  border-top-left-radius: 48rpx;
  border-top-right-radius: 48rpx;
  z-index: 4;
}
```

### 10.2 设计参考
- **风格**：Glassmorphism（毛玻璃）+ Dimensional Layering（层次叠加）
- **来源**：ui-ux-pro-max skill 推荐
- **特点**：半透明模糊、圆角、阴影层次

### 10.3 优化前后对比

| 项目 | 优化前 | 优化后 |
|------|--------|--------|
| 海报高度 | 100vh（全屏） | 50vh（半屏） |
| 过渡方式 | 无过渡，直接跳到商品 | 毛玻璃 + 圆角叠加 |
| 商品区标题 | 无 | 「在销商品」+ 红色装饰线 |
| 视觉层次 | 扁平 | 三层叠加（海报→毛玻璃→商品） |
| 页面背景 | #F5F9FA（灰色） | #ffffff（纯白） |

## 十六、项目阶段划分（项目级）

### 16.1 阶段划分逻辑

本项目按“闭环能力”而不是按页面数量划分阶段。

也就是说，不看“做了多少页”，而看：

1. 是否形成一个最小可运行闭环
2. 是否依赖外部条件
3. 是否具备下一阶段所需的管理能力
4. 是否能进入真实运营

### 16.2 建议阶段

#### 阶段 0：目标与范围确认

- 明确卖什么
- 明确服务谁
- 明确最小闭环

#### 阶段 1：C 端本地闭环

- 商品浏览
- 购物车
- 确认订单
- 地址管理
- 订单列表 / 详情

#### 阶段 2：数据契约与云准备

- 本地数据结构冻结
- 云函数输入输出对齐
- 状态流转对齐

#### 阶段 3：商家后台 MVP

- 订单列表
- 订单详情
- 订单状态处理
- 商品管理

#### 阶段 4：联调闭环

- C 端 + 云端 + 后台打通
- 关键流程联调

#### 阶段 5：上线与运营

- 支付
- 真实环境
- 运营与异常处理

### 16.3 当前所处阶段

当前整体更接近：

- 阶段 1 末尾
- 阶段 2 前面

也就是：

- C 端本地闭环已经比较完整
- 但云端、后台、联调、真实运营还未完成

### 16.4 粗略完成度估算

按不同目标看，完成度不同：

- 本地可演示版：约 80%
- 可真实运营的一期产品：约 40%
- 含后台 + 云端 + 联调的完整工程：约 30% ~ 35%

## 十八、当前协作与 AI 编排共识

### 18.1 当前地图已升级为六层

当前不再只用“五层地图”，而是使用“六层地图”定位问题：

1. 任务层：当前做什么
2. 工程层：模块与边界怎么组织
3. 数据层：字段、状态、契约如何流转
4. 交付层：如何验收、排查、回退、上线
5. 学习层：术语、方法、认知怎么补
6. AI 编排层：模型、上下文、线程、agent、外部记忆怎么配合

### 18.2 当前默认协作原则

- 小闭环：一次只推进一个最小可验证闭环
- 外化：规则、状态、交接写入文档
- 先设计：先对齐目标、范围、依赖、验收，再编码
- 关键节点留档：目标确认、设计确认、编码前、暂停前必须留档
- 控制上下文污染：不同目标尽量拆到不同线程 / 窗口

### 18.3 默认 Bug 排查方式

默认不从“全项目通扫”开始，而按下面顺序：

1. 明确症状与复现步骤
2. 缩小到页面 / 模块 / 数据 / 云函数级
3. 检查局部链路
4. 最后才做全局扫描

### 18.4 当前最近的合理推进顺序

1. 开发者工具人工回归
2. 云端契约对齐设计
3. 最小可观测性补强

说明：

- 这三项比继续堆低价值功能更优先
- 云端接入前，要先把本地闭环和文档闭环做稳
