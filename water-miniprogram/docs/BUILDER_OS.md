# Builder OS 总纲

## 这份文件是什么

这不是项目需求文档，也不是单纯的学习笔记。

它是：

- AI 时代个人 Builder 的操作系统总纲
- 用来统一“认知地图、推进方法、AI 编排、工程落地、成长路线”的总入口

一句话：

**它解决的是“如何稳定地把想法做成系统”这个总问题。**

---

## 一、总目标

当前的核心目标不是：

- 只会让 AI 写代码
- 只会做漂亮页面
- 只会聊方法论

而是：

**成为一个能借助 AI，稳定做出可上线、可维护、可持续运行产品的 Builder。**

这意味着，Builder 的工作已经不只是“实现”。

今天更像是在同时做五件事：

1. 定义问题
2. 拆解任务
3. 组织系统
4. 编排 AI
5. 管理交付与演进

---

## 二、Builder OS 的三层结构

### 第一层：问题地图

用于回答：

- 我现在问的这个问题属于哪一类？

当前采用六层地图：

1. 任务层
2. 工程层
3. 数据层
4. 交付层
5. 学习层
6. AI 编排层

它的作用是：

- 降复杂度
- 防止混层
- 帮助收敛

### 第二层：推进方法

用于回答：

- 一轮任务到底该怎么推进？

当前默认方法是：

- Builder 最小闭环六步法

即：

1. 先定义当前意图
2. 切出最小闭环
3. 先设计，再实现
4. 写清这轮做什么 / 不做什么
5. 关键节点留档
6. 控制上下文污染

### 第三层：人成长地图

用于回答：

- 作为 Builder，本人要长出哪几块能力？

当前四大能力：

1. AI 编排能力
2. 工程能力
3. 产品判断能力
4. 商业闭环能力

---

## 三、为什么这套 OS 在今天重要

因为 AI 时代有一个核心变化：

- **实现速度被大幅放大**

而一旦实现速度被放大，上游和下游都会被放大：

- 上游：问题定义、边界、取舍、任务排序
- 下游：验收、可观测、排查、回退、运维

所以今天真正更稀缺的不是“多会写一点”，而是：

- 判断力
- 结构力
- 裁决力
- 编排力

---

## 四、Builder 的默认工作主链路

默认主链路不是：

- 想到什么做什么

而是：

1. 先判断问题属于哪一层
2. 再定义这轮当前意图
3. 再切最小闭环
4. 再做设计
5. 再决定是否需要线程 / 工作树 / 多 agent
6. 再编码
7. 再验证
8. 再留档

这条主链路的意义不是变慢，而是：

- 降返工
- 降污染
- 降误判
- 提高 AI 协作质量

---

## 五、Builder OS 的几个核心原则

### 1. 小闭环优先于大愿景

大愿景可以保留，但执行必须切成最小可验证闭环。

### 2. 设计优先于实现

先想清楚边界、状态、契约、验收，再让 AI 或自己去写。

### 3. 外化优先于记忆

规则、状态、交接都优先写到文件，不依赖聊天历史和当场状态。

### 4. 局部诊断优先于全局扫描

默认查 bug 不从“AI 全项目过一遍”开始，而是从症状和局部链路开始。

### 5. 编排优先于堆 agent

不是 agent 越多越高级，而是看是否有清楚边界、并行收益和验收方式。

---

## 六、Builder OS 在送水小程序里的对应关系

### 当前问题地图

- 多地址管理：任务层 / 工程层 / 数据层
- 云端接入：工程层 / 数据层 / 交付层 / AI 编排层
- 可观测性：交付层
- 如何开新窗口、切线程：AI 编排层
- “string 是啥”：学习层

### 当前推进方法

- 本地闭环先行
- 云端高冲突主题后置
- 先设计后实现
- 关键节点写入 `CURRENT_TASK` / `HANDOFF_LOG`

### 当前人成长重点

- AI 编排能力：正在迅速建立
- 产品判断能力：已开始形成“意图闭环最短”的视角
- 工程能力：正在从功能实现走向系统成立
- 商业闭环能力：暂缓深挖，但已开始意识到它迟早是核心

## 八、Builder OS 不是终稿

这套系统不是终极真理，而是当前阶段的最优作战图。

未来它大概率还会继续扩展出：

- 更完整的商业层
- 更完整的运营层
- 更完整的多 agent 协作层
- 更完整的评估与自动化层

所以对它最好的态度不是“背诵”，而是：

- 先用
- 再迭代
- 再留下新规则

---

## 九、一句话总结

**Builder OS 的本质，不是教你怎么更快敲代码，而是让你在 AI 时代，更稳地组织问题、任务、系统、协作和成长。**
