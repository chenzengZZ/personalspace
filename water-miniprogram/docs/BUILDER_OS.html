六层地图 / Problem Map

先判断问题属于哪一层，再决定怎么推进。地图不是世界真相，而是你对抗复杂度的作战图。

任务层

现在最该干什么

判断当前目标、拆分顺序、并行还是串行，以及这一轮到底该推进到哪一步。

工程层

系统怎么组织

模块边界、职责切分、公共规则和高冲突主题隔离，都属于工程层。

数据层

信息如何流动

字段、状态、契约、兼容与本地/云端口径对齐，是系统成立的底层。

交付层

如何证明能上线

验收、可观测、排查、回退、恢复、运维意识，让产品从 demo 走向真实运行。

学习层

你还缺什么认知

术语、方法、框架、认知补课与刻意练习，都属于学习层。

AI 编排层

怎么组织 AI 干活

模型选择、上下文控制、线程切分、单/多 agent、skill/MCP/RAG 配合，都是新世代 Builder 的必修课。

推进引擎 / Min-Loop Engine

默认推进不靠激情，不靠“继续做一点”，而是靠最小闭环六步法，让每一轮都真的形成可验证结果。

1

定义当前意图

先说清楚这一轮到底在解决什么，别直接跳进实现。

2

切出最小闭环

功能不是越多越好，关键是核心意图能否最短闭环成立。

3

先设计，再实现

页面关系、数据结构、状态流转、边界情况、验收标准都先想清楚。

4

写清做什么 / 不做什么

没有“不做项”的任务，通常就是边界不清的任务。

5

关键节点留档

目标确认、设计确认、编码前、暂停前都要留下能交接的痕迹。

6

控制上下文污染

新 idea 先归档，高冲突主题拆开推进，讨论窗口不要顺手改项目。

“最小闭环不是小，而是准；先设计不是慢，而是省返工。”

四大能力 / Capability Matrix

地图回答“问题属于哪层”，能力矩阵回答“你本人还要长出哪几块能力”。

AI 编排

组织 AI，不是只调用 AI

  • 模型与强度选择
  • 上下文和窗口控制
  • 线程 / agent / 外部记忆协同
工程能力

让系统成立，不只是让功能可见

  • 模块边界
  • 数据结构与状态流转
  • 可观测、可回退、可维护
产品判断

定义骨架，而不只是挑细节

  • 意图闭环最短
  • 最小成立骨架
  • 优先级与取舍
商业闭环

价值交换能否持续发生

  • 履约与复购
  • 渠道与运营
  • 长期可持续运行

路由决策 / Routing Decisions

很多混乱不是出在代码，而是出在什么时候换线程、换窗口、上多 agent、做全局扫描。

线程 / 窗口

什么时候换线程，什么时候开新窗口

  • 换目标，优先换线程
  • 上下文开始影响判断时，再开新窗口
  • 方法论窗口默认不碰项目逻辑
单 agent / 多 agent

边界不清，先别多 agent

  • 目标清楚
  • 子任务可拆
  • 边界不重叠
  • 结果可分别验收
Bug 排查

先急诊，后体检

  • 先定症状
  • 再缩范围
  • 再查局部链路
  • 最后才全局扫描
新 idea

先归档，再裁决

  • 它是什么
  • 解决什么
  • 和当前项目什么关系
  • 是探索还是准备开工

交付与排查 / Delivery & Observability

交付层不是形式主义，而是让系统在真实世界里出问题时不至于全黑。

最小可观测性

先做到“出问题时不全黑”

  • 关键业务动作是否发生
  • 失败发生在哪一段
  • 用户提示与开发排查信息是否一致
  • 本地与云端口径是否一致

  2. 云端契约对齐设计
  3. 最小可观测性补强
