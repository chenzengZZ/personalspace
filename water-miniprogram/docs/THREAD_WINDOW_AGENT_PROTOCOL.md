# 线程 / 窗口 / Agent 协议

## 这份文件解决什么问题

很多混乱，不是出在代码，而是出在：

- 什么时候该开新窗口
- 什么时候该留在当前线程
- 什么时候该单 agent
- 什么时候才值得上多 agent

这份协议就是为了解决这些问题。

---

## 一、核心对象分别是什么

### 1. 线程 Thread

线程代表：

- 一个明确目标
- 一条连续推进链路

线程回答的是：

- **这轮到底在解决什么？**

### 2. 窗口 Window

窗口代表：

- 当前承载这条线程的上下文容器

窗口回答的是：

- **这轮在哪个上下文里推进最稳？**

### 3. Agent

Agent 代表：

- 一个执行体
- 在明确目标、工具、反馈、边界内推进任务的 AI 单元

它回答的是：

- **这件事由谁来执行，执行方式怎么组织？**

---

## 二、什么时候继续留在当前线程

满足下面条件时，优先留在当前线程：

- 还是同一个目标
- 只是局部深化
- 不需要切换认知模式
- 不会明显增加上下文污染

例子：

- 继续细化确认订单体验
- 继续补同一链路的设计

---

## 三、什么时候开新线程

出现下面情况时，考虑新线程：

- 目标已经换了
- 依赖关系已经换了
- 风险层级明显不同
- 会让当前主链路失焦

例子：

- 从“本地地址闭环”切到“云端契约对齐”
- 从“送水小程序开发”切到“新的 agent 项目”

一句话：

- **换目标，优先换线程。**

---

## 四、什么时候开新窗口

新窗口不是为了新鲜，而是为了：

- 隔离上下文
- 提升稳定性
- 避免污染

适合开新窗口的情况：

- 讨论方法论，已经明显压住项目推进
- 想开一个完全不同方向的新项目
- 当前窗口背景负载已经太高
- 当前窗口里已经混入太多不同层问题

不适合开新窗口的情况：

- 只是还没想清楚
- 只是轻微发散
- 只是想“保险一点”

一句话：

- **上下文开始影响判断时，再开新窗口。**

---

## 五、什么时候默认单 agent

默认单 agent 的情况其实很多：

- 目标还没定清
- 还在设计阶段
- 强依赖顺序
- 边界没切开
- 你自己都还没想明白验收标准

这时多 agent 不会更高级，只会更乱。

---

## 六、什么时候值得上多 agent

必须同时满足下面四条，才认真考虑：

1. 目标清楚
2. 子任务可拆
3. 边界不重叠
4. 结果可分别验收

如果少一条，就先别上。

---

## 七、送水小程序里的典型例子

### 适合单线程 / 单 agent

- 明确确认订单页体验优化
- 设计多地址规则
- 讨论最小可观测性

### 适合新线程

- 云端接入
- 商家后台 MVP
- 新 agent 项目探索

### 未来可能适合多 agent

- 一个 agent 看云函数契约
- 一个 agent 看前端页面承接
- 一个 agent 整理交付 checklist

前提是：

- 写文件边界清楚
- 任务彼此不抢同一核心文件

---

## 八、开新线程前最少要写清什么

至少四件事：

1. 它是什么
2. 它解决什么问题
3. 它和当前项目什么关系
4. 它现在是探索中，还是准备开工

---

## 九、一句话口诀

**换目标，开新线程；换容器，看上下文；边界不清，先别多 agent。**
