Decision Atlas

当问题很多时，
先裁决，再推进。

这不是知识点陈列，而是常用判断的快速图谱。它解决的不是“知道更多”，而是“现在最该怎么做”。

Coding Gate

我现在能开始写代码吗？

目标清楚吗？

如果这轮到底解决什么都说不清，先别写。

最小闭环切出来了吗？

如果任务还是大而全，就先缩小到可验证闭环。

边界和验收标准清楚吗？

页面、数据、状态、风险和“不做项”都清楚了，才进入实现。

结论

只要前三条有一条不清，默认继续设计；三条都清楚，才写代码。

Thread Gate

我要开新线程吗？

目标换了吗？

如果已经不是同一个明确目标，优先开新线程。

风险层级换了吗？

从本地闭环切到云端接入、支付、后台，通常值得新线程。

会不会让当前主链路失焦？

如果会，就不要硬塞进当前线程。

结论

换目标，优先换线程；没换目标，只是局部深化，就留在线程内。

Window Gate

我要开新窗口吗？

当前上下文已经开始压判断了吗？

如果明显发散、混层、负载过高，开新窗口隔离上下文。

这是元认知还是执行问题？

方法论讨论压住项目推进时，拆到新窗口更稳。

是否需要完全不同的工作模式？

执行窗口和元认知窗口不要长期混用。

结论

不是一发散就开新窗口，而是当上下文已经影响质量时再开。

Agent Gate

现在值得上多 agent 吗？

目标清楚了吗？

目标不清，多 agent 只会并行制造混乱。

任务切开了吗？

子任务必须可拆、边界清楚、写集不重叠。

结果能分别验收吗？

没有独立验收标准，就不适合并行。

结论

单 agent 是默认，只有“清楚、可拆、低冲突、可验收”才值得多 agent。

Debug Gate

查 bug 要不要全局扫描？

症状清楚吗？

预期、实际、复现步骤不清，就先别通扫。

范围能缩吗？

先缩到页面、模块、数据或云函数级。

局部链路查了吗？

先急诊，后体检。局部查不出来，再扩大。

结论

默认排查顺序：先定症状，再缩范围，再查链路，最后全扫。

Idea Gate

新的 idea 要立刻做吗？

它和当前目标同一件事吗？

如果不是，先别挤进当前链路。

它是探索还是准备开工？

很多好 idea 现在只该归档，不该立刻执行。

它有最小闭环吗？

没有闭环定义，就暂时别消耗执行资源。

结论

新 idea 默认先归档，再裁决；不是新想法一来就开工。

第一原则

先裁决，再推进；先收边界，再开工。

第二原则

真正浪费的不是慢，而是方向不准还高速推进。

第三原则

Builder 的强大，不是知道更多，而是更会决定现在不做什么。
