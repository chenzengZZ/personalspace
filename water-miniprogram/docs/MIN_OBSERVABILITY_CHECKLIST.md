# 小程序最小可观测性清单

## 这份文件的作用

这不是要一步到位做成完整监控平台。

它解决的是更现实的问题：

- 明天真的开始接云或联调时，出了问题至少知道先看哪里
- 不把“可观测性”误解成一个很大、很远的工程

一句话：

**先把“出问题时不至于全黑”做出来。**

---

## 一、最小可观测性到底要看到什么

至少要能看见三类东西：

1. 关键业务动作发生了没有
2. 失败发生在哪一段
3. 用户看到的提示和开发排查信息是不是同一件事

---

## 二、送水小程序最小版应该观察哪些点

### 1. 下单链路

至少要知道：

- 进入确认订单页
- 是否有结算商品
- 是否已选地址
- 点击提交订单
- 提交成功 / 失败
- 返回了什么错误信息

### 2. 地址链路

至少要知道：

- 地址列表是否成功加载
- 新增 / 编辑 / 删除 / 设默认是否成功
- 删除默认地址后兜底规则是否触发

### 3. 订单链路

至少要知道：

- 订单列表是否加载成功
- 搜索 / 筛选是否生效
- 状态流转是否成功
- 订单详情是否能取到对应订单

### 4. 云函数链路

至少要知道：

- 调的是哪个函数
- 传了什么核心参数
- 返回 code 是多少
- 失败发生在调用前、调用中，还是返回后

---

## 三、最小实现不要追求什么

当前不必一上来追求：

- 全量埋点平台
- 自动告警系统
- 完整监控面板
- 指标大盘

先做：

- 基础日志
- 统一错误返回
- 关键节点标记
- 最小排查路径

---

## 四、最小清单

### A. 统一错误口径

- 用户提示是什么
- 开发排查信息是什么
- 是否有统一 `code / message / data`

### B. 关键业务节点有日志

- 页面进入
- 关键按钮点击
- 云函数调用
- 成功 / 失败结果

### C. 本地与云端口径一致

- 本地 mock 成功是什么样
- 云端真实成功是什么样
- 失败返回结构是否一致

### D. 排查路径清楚

出了问题时，至少知道先查：

1. 页面状态
2. 本地缓存 / storage
3. 调用参数
4. 云函数返回
5. 用户提示

---

## 五、什么叫“已经够用了”

对于当前阶段，满足下面几点就算“最小够用”：

- 下单失败时，不再只知道“失败了”
- 地址异常时，能快速判断是页面、存储还是调用问题
- 云函数失败时，能区分调用失败和业务失败
- 人工回归时，有明确链路可照着查

---

## 六、明天如果真做，推荐顺序

1. 先统一 `utils/cloud.js` 的成功 / 失败口径
2. 再给关键链路加最小日志点
3. 再补一份“排查顺序”
4. 最后再考虑更完整的监控或埋点

---

## 七、一句话

**可观测性不是“看得见代码”，而是“出问题时，看得见系统刚刚发生了什么”。**
