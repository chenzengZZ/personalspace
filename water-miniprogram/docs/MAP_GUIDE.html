送水小程序训练手册

这不是一张“看完就关掉”的说明页，而是一份反复练习用的项目导航图。

你后面每次卡住，都先回来问自己：我这个问题属于哪一层？现在是在设计、拆分、实现，还是在交接？

用途：定位问题 用途：训练判断 用途：协作交接 用途：避免线程污染

一、六层地图

任务层

你在问什么

  • 当前目标是什么
  • 任务怎么拆分
  • 哪些能并行，哪些必须串行
  • 要不要新线程 / 多 agent
工程层

你在问什么

  • 模块怎么分
  • 公共规则放哪里
  • 边界和职责是否清楚
  • 要不要新工作树 / 分支
数据层

你在问什么

  • 字段长什么样
  • 状态怎么流转
  • 前后端契约是否一致
  • 兼容是否稳定
交付层

你在问什么

  • 怎么验收
  • 怎么回退
  • 怎么排查
  • 上线前检查什么
学习层

你在问什么

  • 当前最该补哪些术语
  • 这个问题属于地图哪一层
  • 怎么保持最小闭环
  • 什么时候该收敛
AI 编排层

你在问什么

  • 该用什么模型 / 思考强度
  • 什么时候切新窗口或新线程
  • 什么时候适合多 agent
  • skill / MCP / RAG / 外部记忆怎么配合
训练动作：以后每次提问，先尝试给它归类到“任务层 / 工程层 / 数据层 / 交付层 / 学习层 / AI 编排层”之一。归不清也没关系，但必须先尝试。

二、系统关系图

现实目标（稳定上线、长期运维）
└─ Thread 线程：一个明确任务
   ├─ 规划多地址管理
   ├─ 接云环境
   └─ 做上线清单
      └─ Agent：执行者
         ├─ 查数据契约
         ├─ 看 UI / 交互
         └─ 整理文档 / 风险

Repo 仓库：water-miniprogram
└─ Branch 分支：代码历史路线
   ├─ main
   ├─ feat/cloud
   └─ feat/admin
      └─ Worktree 工作树：独立工位
         ├─ wt-main-stable
         ├─ wt-cloud
         └─ wt-ui
一句话记忆
  • 线程是任务
  • 工作树是工位
  • 分支是历史路线
  • agent 是执行者
最常见误区
  • 把线程当成分支
  • 把工作树当成必须一开始就开的东西
  • 以为多 agent 越多越快
  • 以为代码实现比任务拆分更早

三、默认工作顺序

正确顺序
  1. 对齐目标
  2. 拆分任务
  3. 分析依赖与冲突
  4. 决定线程 / 工作树 / agent
  5. 明确范围
  6. 明确验收标准
  7. 留档
  8. 最后才写代码
当前项目提醒
  • 云端接入是高冲突主题
  • 地址结构、订单流转属于高风险
  • 样式和静态文案属于低风险
  • 当前更适合先设计再进入实现

四、窗口分工与新 idea 规则

窗口分工
  • 执行窗口：推进项目设计、开发、排查、交付
  • 元认知窗口：讨论方法论、学习、协作规则、认知地图
  • 讨论窗口默认不改项目文件
  • 执行窗口优先保证当前链路不断裂
新 idea 处理
  • 先记下来，不立刻插入当前开发链路
  • 先判断它属于当前项目，还是新线程
  • 至少说明：它是什么、解决什么、和当前项目什么关系
  • 再决定是归档、排期，还是单独启动
核心原则：新想法不等于马上开工。先归档，再裁决；先分流，再执行。

五、关键环节留档规则

必须留档的时刻
  • 目标确认后
  • 关键设计确认后
  • 开始编码前
  • 暂停前
  • 切线程前
  • 开新窗口前
对应文件
  • docs/CURRENT_TASK.md：当前任务状态
  • docs/HANDOFF_LOG.md：暂停与交接
  • docs/WORKFLOW_CHECKLIST.md：协作规则
  • PROJECT.md：项目关键决策
  • CLOUD_HANDOFF.md：云端契约说明
训练动作：以后每完成一个关键节点，不要问“要不要记下来”，而是直接问“这次应该更新哪个文件”。

六、自检清单

问题自检
  • 我这个问题属于哪一层？
  • 这是设计问题还是实现问题？
  • 是否会碰核心链路？
  • 是否依赖云端或外部条件？
任务自检
  • 这一轮只做什么？
  • 这一轮明确不做什么？
  • 验收标准是什么？
  • 是否需要留档？
实现前自检
  • 页面关系清楚了吗？
  • 数据结构清楚了吗？
  • 边界情况想过了吗？
  • 现在真到了写代码的时候吗？
快速裁决器：如果一个问题同时碰到“核心链路 + 云端依赖 + 数据结构”，先别写代码，先做设计确认。

七、卓越 builder 的训练方向

不是只会做页
  • 能定义最小成立骨架
  • 能判断先做什么、后做什么
  • 能看见产品的价值交换和约束
  • 能把经验沉淀成规则与交接
当前最该补
  • 骨架判断，而不是先磨 UI
  • 优先级裁决，而不是一路展开
  • 上线与运维意识，而不是只看能跑
  • 留档与复盘，而不是只靠当场记忆
  • AI 编排能力，而不是只会单次提问
你要练的不是“让 AI 多写一点”，而是“让系统更稳地成立、推进、交接、上线”。

八、刻意练习方法

日常练法
  1. 先归类问题
  2. 再决定当前阶段
  3. 再问最关键的一个问题
  4. 最后才考虑实现
适合你的节奏
  • 允许自由发散
  • 但每轮最后必须收敛
  • 一次只推进一个最小闭环
  • 把记忆外化，不依赖聊天历史

建议配合 `docs/TERMS_MINI.md` 和 `docs/CURRENT_TASK.md` 一起使用。

九、Builder 最小闭环六步法

六步法
  1. 先定义当前意图
  2. 切出最小闭环
  3. 先设计，再实现
  4. 写清这轮做什么 / 不做什么
  5. 关键节点留档
  6. 控制上下文污染
核心提醒
  • 最小闭环不是小，而是准
  • 先设计不是慢，而是省返工
  • 留档不是形式，而是协作记忆系统
  • 污染先伤判断，再伤实现
这套方法不是“多一个流程”，而是为了防止 AI 协作时看起来很忙、实际上没有形成闭环。

十、AI 时代 Builder 四大能力

AI 编排
  • 模型 / 强度选择
  • 上下文控制
  • 线程、agent、外部记忆配合
工程能力
  • 模块边界
  • 数据结构与状态流转
  • 可观测、可排查、可维护
产品判断
  • 意图闭环最短
  • 最小成立骨架
  • 优先级与取舍
商业闭环
  • 价值交换
  • 履约与复购
  • 长期可持续运行
问题地图回答“这个问题属于哪层”，四大能力回答“你本人要长出哪几块能力”。两个都要有。
