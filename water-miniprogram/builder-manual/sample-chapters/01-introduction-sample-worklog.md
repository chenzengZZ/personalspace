# 导言样章工作流记录

## 0. 入口校验

- `BOOK_BRIEF.md` 已将导言固定为第一优先样章。
- `CHAPTER_WORKFLOW.md` 已固定单章流程：立题 → 定命题与边界 → 收证据 → 写提纲 → 写初稿 → reverse outline → 清稿 → 硬验收 → 请求并书。
- `CHAPTER_CONTRACTS.md` 已明确导言职责：定义这本书为什么存在、主读者 / 非读者、边界与阅读方式。
- 当前样章目标已被固定为：验证“读者为什么值得继续读下去，而不是把它当成又一份结构化讲义”。

结论：入口条件满足，可以进入导言样章执行。

## 1. 章节问题陈述

这章要解决的问题不是“怎么介绍这本书”，而是更尖锐的一件事：

**一个已经能用 AI 快速做出 demo 的读者，为什么还需要继续读这本书？**

如果导言不能让他感到——自己现在卡住的根本问题，不是工具不够多，而是目标、边界、上下文与交付正在一起失控——那么这本书就会被误读成另一份整理得更好的 AI 讲义。

## 2. 章节中心命题

**AI 让实现变便宜之后，Builder 真正稀缺的能力变成了：把一个值得做的东西组织成能持续运行的系统；这本书存在的意义，就是把读者从“和 AI 零散聊天”带到“用 AI 稳定推进真实交付”的工作方式。**

## 3. 章节非目标 / 不写什么

- 不完整展开 `目标篇` 的价值系统，只把问题送到“Builder 到底在追求什么”。
- 不提前证明 `原理篇` 的不变量，只在导言层面点出“为什么今天必须重新讨论 Builder”。
- 不写工具清单、prompt 套装、平台比较或框架教程。
- 不写单一项目案例复盘。
- 不讨论传播版、工程运行时版或网页壳。

## 4. 证据包

### 4.1 文件证据

1. `WHY_CURRENT_VERSION_IS_3_OF_10.md`
   - 当前核心问题不是网页承载，而是文本只解决了“能读下去”，没有解决“想继续读下去”。
   - 导言被点名为第一优先重写样章，因为它决定读者愿不愿继续读。
   - 高质量样章必须开头打痛问题、只有一个中心命题、抽象判断要被场景托住、要推进认知、要留下可记住的句子。

2. `BOOK_BRIEF.md`
   - 主读者是：有真实构建意愿与交付压力，但 AI 使用仍主要停留在 chatbot 层、尚未形成 agent / workflow / system 级编排能力的人。
   - 唯一承诺是：把和 AI 的零散聊天升级成能定目标、拆最小闭环、组织上下文、推进交付的工作方式，减少返工、污染和误判。
   - 导言是第一优先样章，因为它最直接验证“读者为什么值得继续读”。

3. `CHAPTER_CONTRACTS.md`
   - 导言只负责：定义本书为什么存在、写给谁、边界是什么、怎么读。
   - 导言不负责：完整展开价值目标、不变量、能力结构、运行协议、案例细节、术语索引。

4. `LEARNING_BOOK_DRAFT.md`
   - `33-44`：现有稿已明确主读者 / 非读者边界。
   - `75`：本书被定义为帮助个体 Builder 抓住稳定结构的地图。
   - `137-141`：现有稿已给出现实承诺：建立稳定构建框架，减少返工、污染和误判，把 AI 从随机输出源变成可编排杠杆。
   - `153-159` 与 `274`：Builder 的目标不是多产物，而是让价值在系统中持续发生。
   - `476`：AI 编排能力更像“组织智能系统”的能力，而不是“和模型聊天”的能力。
   - `638-648`：个体 Builder / 小团队 Builder 没有大组织缓冲，更容易被注意力、时间、上下文承载限制拖住。
   - `672-705`：上下文如果不被组织，系统会出现污染、过载和失焦；问题先归位，讨论和实现才不会互相污染。

5. `LEARNING_TEXT_EDIT_RULES.md`
   - 导言前台主体应强化“这本书在回答什么”和“怎么读”，但不能暴露后台脚手架语言。

### 4.2 推断

- 开头必须先落到一个读者已经经历过的失败现场：AI 把 demo 拉起来了，但目标、上下文、版本、边界和交付一起失控。
- 导言必须采用更直接的第二人称与场景压力，而不能继续停留在概念说明层。
- 导言要明确告诉读者：这本书不是来帮你“多做一点”，而是帮你“别在加速中失控”。

### 4.3 待验证

- 样章标题是否最终保留为“导言”，还是改成更有抓力的命名，留待总控审。
- 导言里对全书七篇的摘要是否已经足够，还是仍然略多，留待总控审。

## 5. 章节提纲

1. 用一个高度可识别的失败现场开篇：demo 很快做出来，但系统开始失控。
2. 指出问题迁移：AI 没有替人解决判断、边界、上下文和交付，只是把这些问题提前暴露。
3. 收束主读者 / 非读者：给正在被加速的人看，不给只想抄 prompt 或学单一工具的人看。
4. 定义 Builder：不是职位，而是把问题、价值、AI 与系统组织起来的人。
5. 说明承诺与边界：不给答案库，给工作方式；不给工具教程，给稳定地图。
6. 说明阅读方式，并把问题自然交给下一章“目标篇”。

## 7. Reverse Outline 记录

1. `开场四段`
   - 功能：先把“快，但失控”的问题打痛，阻止导言一上来滑进概念定义。
   - 判断：有效，是整章的起势段。

2. `第一节：你已经见过问题，只是还没把它叫对名字`
   - 功能：把“实现门槛下降”重新解释为“组织难题提前暴露”。
   - 判断：承担整章中心命题，是认知转折点。

3. `第二节：这本书写给正在被加速的人`
   - 功能：筛选读者，把“是否会写代码”从硬门槛移开，改为“是否承受真实交付压力”。
   - 判断：必要，且直接承接 `BOOK_BRIEF.md`。

4. `第三节：Builder 不是职位，是一种组织责任`
   - 功能：给出身份定义，但不越界展开目标篇与能力篇。
   - 判断：必要，但初稿里有一句偏抽象，清稿时压缩。

5. `第四节：这本书给你的，不是答案库，而是工作方式`
   - 功能：交代唯一承诺与非承诺，说明这本书和讲义 / 教程 / prompt 手册的差别。
   - 判断：是整章收束前的核心节，需要保留高压强句子。

6. `第五节：把它当地图，不当教程`
   - 功能：说明阅读方式，并把问题交给下一章。
   - 判断：有必要，但不能写成整书摘要，所以只保留最低限度的七篇指向。

### 8.1 本轮二改（按总控要求）

- 只处理两件事：语气更锋利、更像真人；命题句、结论句、收束句更短、更硬、更清楚。
- 修改范围只收在第 `4` 节与第 `5` 节，不改中心命题，不扩写结构，不新增摘要段。
- 第 `4` 节把“承诺 / 边界”改成更直接的拒绝式表达，降低说明书口气，提高读者压强。
- 第 `5` 节把原来的整书路径说明压缩成一段短句，减少摘要感，强化“地图而非教程”的使用方式。

  - “把它当答案库来读，你会嫌它慢。”
  - “你现在不缺动作。你先缺终点。”

- 本轮未改中心命题，未扩结构，未新增大段摘要。

### 8.3 本轮 v3（按总控新要求）

- 主读者外扩：不再默认读者来自互联网 / 产品 / 运营 / 技术行业，把老师、快消、传统行业、单位体系、非 IT 背景一起纳入合法读者样本。
- 主读者共同点改写为：手上有真实问题，想把问题变成能用结果，但没受过系统产品 / 工程训练，AI 目前主要仍停在聊天和生成层。
- 开头场景去互联网默认语境：删去 `PRD / demo / 接口 / 上线` 这类默认圈内词，换成跨行业都能代入的“方案 / 通知 / 课件 / 流程 / 材料 / 初稿”。
- 去翻译腔，但不生硬清洗术语：保留 `Builder`，但第一次出现时先用人话接住；减少 `构建` 一类前台读感偏翻译腔的表达，改成“把事做成”“变成能用结果”“做出来并推到真实使用里”。
- 文风再坏一档：重点打击错法、幻觉和“热闹像进展”的假象，不嘲讽读者本人。
- 关键句继续收短、收硬：
  - “问题不是做不出来。问题是东西越出越多，事反而越来越做不成。”
  - “当生成变便宜，失控就会变贵。”
  - “它教你怎样别被速度耍得团团转。”

## 9. 硬验收

1. `问题成立`：通过  
   - 开头先打出“做得更快，却更失控”的失败现场。

2. `命题唯一`：通过  
   - 整章只服务一个命题：这本书的价值在于帮助读者从零散聊天升级为稳定推进交付的工作方式。

3. `证据托底`：通过  
   - 主读者、承诺、个体 Builder 约束、上下文污染与 3/10 诊断均有文件证据支撑。

4. `认知推进`：通过  
   - 读者会从“我缺更多工具”被推进到“我缺组织目标、上下文与交付的方法”。

5. `句子可记住`：通过  
   - 至少保留三句能钉住整章的句子。

6. `节奏完整`：通过  
   - 起势（失败现场）→ 推进（问题迁移）→ 收束（承诺与边界）→ 下启（阅读路径与下一章）。

7. `讲义腔下降`：通过  
   - 清稿后明显减少模板式判断句堆叠，增加场景段与直接对话感。

8. `桥接腔下降`：通过  
   - 不暴露“本篇解决 / 下启 / 不处理”这类后台脚手架语言。

9. `范围不漂移`：通过  
   - 未展开目标篇、原理篇、能力篇、运行篇、案例篇正文职责。

10. `作者可用`：通过  
   - 这版导言可直接用于测试“读者是否愿意继续读”，而不只是验证结构是否完整。
