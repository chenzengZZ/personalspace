# 第六篇：案例篇

## 职责

- 用真实项目验证整本书的框架
- 证明案例是验证器，而不是定义器

## 开篇命题句

案例的价值不在于提供标准答案，而在于验证同一套构建框架如何穿过不同项目的表面差异。

## 章节桥接卡

- 上承：运行系统已经定义，这一篇要验证那套系统能否穿过不同项目类型而不失真。
- 本篇解决：说明案例为什么只能做验证器，并展示不变量与可变量如何在不同项目中映射。
- 下启：既然案例已经证明框架可用，下一步就要把高频出现的结构沉淀成可调用工具层。
- 不处理：让任何单一案例反向决定整本母本的结构。

## 核心论点 v1

- 案例的作用是验证框架，而不是定义框架
- 单一案例不能反向决定母本结构
- 同一套目标、原理、能力与运行系统，应当能穿过不同项目表面差异
- 案例篇真正要展示的不是“答案”，而是“映射方法”
- 一个好案例，应该同时说明哪些东西不变，哪些东西会变

## 正文 v1

### 6.1 为什么要有案例篇

一本方法论书如果完全没有案例，往往会停留在“结构上成立”，却无法证明自己能穿过现实世界。

但案例篇也有一个天然风险：

- 一旦案例放得太前，它就会反过来定义整本书

所以案例篇必须放在后面。  
它的角色不是成为理论来源，而是成为理论验证器。

这一区分非常重要。因为如果一本书的结构是从单一案例里长出来的，那么它很容易只对那个案例成立。  
而这本手册真正追求的是：先建立通用母本，再看通用母本如何在不同项目里落地。

### 6.2 如何阅读案例篇

案例篇不是让读者抄答案的。

如果读者把案例篇当成：

- “送水小程序怎么做”
- “agent 项目怎么做”
- “某一类产品的标准步骤”

那就读偏了。

案例篇真正要展示的是：

- 同一套目标、原理、能力、运行系统，如何映射到不同类型的项目

所以读案例时应该重点看：

- 它验证了哪一部分母本
- 它没有验证哪一部分
- 在这个案例里，哪些东西是不变的
- 哪些东西是为了适应场景而变化的

这会让案例变成“结构训练器”，而不是“作业答案”。

### 6.3 案例一：交易型履约产品

以交易型履约产品为例，最容易让人误判的地方是：

- 以为这是一个“页面很多的小程序”

但它的业务本质其实更接近：

- 用户发起需求
- 系统承接需求
- 履约链路完成交付
- 状态在多个节点中流转

所以在这类产品里，真正的最小骨架通常不在 UI，而在：

- 商品结构
- 地址结构
- 订单结构
- 状态流转
- 用户的最短下单闭环

这也是为什么像送水小程序这种 case，真正关键的东西往往不是首页做得漂不漂亮，而是：

- 交易链路是否成立
- 地址和订单是否能稳定承接
- 状态是否清楚
- 履约逻辑是否可扩展

这个案例验证了母本中的几条关键判断：

- 价值先于实现
- 最小骨架先于细节
- 产品判断先于 UI 沉迷
- 数据和状态流转是系统中轴

### 6.4 案例二：Agent 原生产品

Agent 原生产品会验证母本的另一条重要通用性：

- 这套书并不只适用于“传统前端 + 后端 + 页面”产品

在 agent 原生产品中，最小骨架通常不再是某几个页面，而更可能是：

- 目标表达方式
- 工具调用边界
- 记忆与状态传递
- 执行回路
- 反馈与纠偏机制

这意味着，表面产品形态虽然变化很大，但母本中的很多核心问题依然成立：

- 目标是否清楚
- 约束是否被纳入
- AI 是否被正确编排
- 运行机制是否能稳定工作

所以 agent 原生产品不是例外，它恰恰能帮助证明这本书的通用性。

### 6.5 案例三：内容 / 工具 / 服务型产品

这类案例的意义在于进一步打破单一产品形态带来的误解。

内容型产品、工具型产品、服务型产品，在表面上可能和交易履约产品完全不同，但它们仍然会面对相似的根本问题：

- 价值是什么
- 最小成立骨架是什么
- 反馈回路在哪里
- 交付如何成立
- 商业闭环如何形成

变化的是：

- 用户路径
- 工程结构
- 数据组织方式
- 交付模式

不变的是：

- 这套母本的主干问题

所以这一类案例的存在，是为了帮助读者确认：

- 这本书讨论的不是某种产品壳子，而是更通用的构建逻辑

### 6.6 对比：不同案例，哪些东西不变

当我们把不同案例放在一起看，会发现有一些东西始终不变：

- Builder 的目标仍然是创造可持续价值系统
- 原理层的不变量仍然有效
- 能力结构仍然成立
- 运行系统仍然需要问题定位、闭环推进、上下文编排、交付保障与节奏管理

换句话说，案例之间真正稳定的，不是页面、部署方式或技术选型，而是：

- 构建活动的逻辑骨架

### 6.7 对比：不同案例，哪些东西会变

与此同时，案例之间也确实有大量变化：

- MVP 的具体形态
- 产品交互路径
- 数据结构
- 商业模式
- 工程形态
- 履约方式

这些变化不是噪音，它们正是案例存在的价值之一。  
因为只有看到可变量，读者才不会误以为母本在提供某种“统一壳子”。

所以案例篇必须同时训练两种能力：

- 看见不变量
- 也看见可变量

### 6.8 案例篇结论

案例篇最终想证明的是：

- 母本不是某个项目的说明书
- 母本是一套可以穿过不同项目表面差异的构建框架

它不是让人复制案例，而是让人通过案例学会：

- 如何把同一套主线映射到不同现实对象上

这才是案例篇的真正价值。

接下来真正需要做的，不是继续堆更多案例，而是把这些案例中反复出现的结构沉淀成术语、模板、清单与协议，这正是附录篇要承担的工作。
